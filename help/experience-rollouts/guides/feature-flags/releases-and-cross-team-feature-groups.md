# 릴리스 및 팀 간 기능 그룹 {#releases-and-cross-team}

릴리스 및 팀 간 기능 그룹 모두 여러 팀 및 애플리케이션에서 조정된 롤아웃을 지원합니다. 올바른 선택은 캐싱 요구 사항, 대상 타기팅 요구 사항 및 프로세스에 대한 제어 정도에 따라 다릅니다.

## 팀 간 기능 그룹 {#cross-team-feature-groups}

팀 간 기능 그룹을 사용하면 서로 다른 팀이 소유한 애플리케이션에서 기능 플래그를 번들로 지정하고 다양한 대상 기준으로 관리할 수 있습니다.

주요 특성:

* **셀프 서비스** — 하나를 만드는 데 필요한 지원 요청이 없습니다.
* **다양한 대상 타깃팅** — 전체 대상 기준 집합(프로필 특성, 컨텍스트 변수, 롤아웃 비율)을 지원합니다.
* 만들 수 있는 숫자의 **제한 없음**
* **캐싱 없음** - 기능 플래그 평가를 수행하려면 API 호출이 필요합니다. SDK 수준 캐싱은 지원되지 않습니다.

단계별 작성 지침은 [팀 간 기능 그룹 만들기](create-cross-team-feature-group.md)를 참조하십시오.

## 릴리스 {#releases}

릴리스는 SDK 수준에서 캐싱이 필요한 크기가 크고 조정된 론치를 위해 설계되었습니다. 릴리스는 인증 토큰 기반 대상 타깃팅을 사용합니다.

주요 특성:

* **지원 요청 필요** - 릴리스가 완전히 셀프 서비스가 아닙니다. 릴리스를 만들려면 경험 롤아웃 지원 센터에 문의하십시오.
* **SDK 캐싱** - 모든 릴리스 데이터가 서버 SDK에서 로컬로 캐시되므로 API 호출이 줄어듭니다
* **제한된 대상 기준** — 대상 규칙은 인증 토큰(국가, 이메일, 사용자 ID, 백분율 등)을 기반으로 합니다.
* **제한된 활성 릴리스** — 한 번에 한정된 수의 활성 릴리스가 존재할 수 있습니다. 3개월 이내에 릴리스를 닫거나 기준선을 지정할 계획입니다

## 비교 {#comparison}

| | 크로스 팀 기능 그룹 | 릴리스 |
|---|---|---|
| 셀프서비스 | ✓ | 지원 요청 필요 |
| 풍부한 대상 기준 | ✓ | 제한적 |
| A/B 테스트 | ✗ | ✗ |
| SDK 캐싱 | ✗ | ✓ |
| 활성 제한 | 제한 없음 | 제한적 |

## 권장 사항 {#recommendation}

사용 사례에 풍부한 대상 타기팅 및 셀프 서비스 관리가 포함된 경우 **팀 간 기능 그룹**&#x200B;을(를) 사용하십시오. 백엔드 서비스에 SDK 수준 캐싱이 필요하고 릴리스의 더 간단한 대상 기준이 충분한 경우 **릴리스**&#x200B;를 사용하십시오.
